4 A Note on Namespaces
5 --------------------
6 
7 All interfaces live within the `Farsyte` namespace, to avoid
8 collisions with simiarly named things defined elsewhere.

10 Header File Overview
11 ====================
12 
13 Farsyte::Utility "utility.h"
14 ----------------
15 
16 The `utility.h` header collects general utility code that
17 is not associated with any of the larger packages within
18 the toolkit project. This could be code that is shared
19 across multiple packages (without an obvious owner), or it
20 could be entire packages that are simply too small to
21 warrant tracking as their own library.
22 
23 Farsyte::Testing "testing.h"
24 ----------------
25 
26 The `testing.h` header provides declarations for the API
27 for the library that is used to record test results in a
28 form that allows an automated build system (currently the
29 Bamboo continuous intergration system from Atlassian) to
30 track testing results.
31 
32 The `Testing` Library implements four classes:
33 
34  * The [Oops](classFarsyte_1_1Testing_1_1Oops.html) class
35  is the base class for exceptions thrown within
36  the `Testing` library representing failures of
37  the library or failure to follow the requirements
38  of the library when calling it.
39 
40  * The [Test](classFarsyte_1_1Testing_1_1Test.html) class
41  represents a single test, counting as one test when presenting
42  counts of tests passed or failed. Tests may indicate one or more
43  conditions that fail or were skipped, with supporting text, as
44  well as errors occurring in the testing process.
45  It is not uncommon for a Test to correspond to a single method
46  name within a class, and for the code to test various aspects of
47  the method in sequence.
48  If two Test objects have the same test name and the same suite
49  name, their results will be combined during reporting.
50 
51  * The [Suite](classFarsyte_1_1Testing_1_1Suite.html) class
52  represents a collection of Test objects that are logically
53  related; tests with the same Suite name are grouped together when
54  reporting summaries of test results.
55  It is not uncommon for a Suite to correspond to a single Class to
56  be tested within a library.
57  If two Suite objects have the same name, their results will be
58  combined during reporting (even if they come from tests in quite
59  different subprojects). Risks associated with this can be
60  mitigated by assuring that class names are distinct between
61  subprojects of a superproject, when tests are run across the
62  entire superproject as a unit.
63 
64  * The [Log](classFarsyte_1_1Testing_1_1Log.html) class
65  corresponds to a single stream of test output produced by
66  sequentially running a series test suites, each of which
67  sequentially runs a series of test cases. Note that the Log class
68  allows test programs to have multiple Log objects open at once,
69  allowing a test program to persue multiple test series in parallel
70  if this is appropriate and useful.

73 Farsyte::Matrix "matrix.h"
74 ---------------
75 
76 Matrix math is provided as a genearlized template for dense
77 rectangular matrices, a derived template for working with column
78 vectors, and a class representing a position in a three dimensional
79 state space (such as distance north, east, and above the flagpole).
80 
81 The `Matrix` Library implements two templates and one class:
82 
83  * The [Matrix](classFarsyte_1_1Matrix_1_1Matrix.html) class
84  template supports dense rectangular matrices with any small
85  positive integer number of rows and columns, containing data
86  elements of any data type that supports the desired operations.
87  This class currently provides class methods retrieving the
88  dimensions of the matrix, and member methods providing the
89  operations listed here. More operations will be added once I am
90  happy with code quality, test coverage, and documentation for
91  these operations:
92  * Initialization
93  * Addition
94  * Difference
95  * Negation
96  * Transpose
97 
98  * The [ColVec](classFarsyte_1_1Matrix_1_1Colvec.html) class template
99  is a proxy class providing the obvious API adjustments to use
100  for column vectors, which are simply a one-column matrix.
101  The API differences mainly involve not having to redundantly
102  specify column numbers when working with a ColVec class.
103 
104  * The [ThreeVec](classFarsyte_1_1Matrix_1_1ThreeVec.html) class
105  likewise provides for a three-element column vector such as might
106  be used to represent a Position (north, east and above the
107  flagpole), Velocity, Direction, and so on.
108  The `ThreeVec` class adds the Cross Product operation, which is
109  not provided for general Matrix and Column Vector objects.
